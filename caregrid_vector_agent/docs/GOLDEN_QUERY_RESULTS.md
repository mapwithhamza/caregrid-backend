# Golden Query Evaluation Results

Stage 14 of the **caregrid_vector_agent** package introduces a curated
suite of 10 *golden queries* that exercise the full
`run_recommendation` pipeline end-to-end. They live in
[`agent_core/demo_queries.py`](../agent_core/demo_queries.py) as
`GOLDEN_QUERIES` and are evaluated automatically by
[`tests/test_golden_queries.py`](../tests/test_golden_queries.py).

The suite serves three purposes:

1. **Regression guard** — each golden query exercises a different
   combination of capability / state / facility_type / trust /
   urgency, so any regression in the intent parser, retriever,
   validator, or scorer is caught on every run of `pytest`.
2. **Behavioural contract** — every query must produce a
   well-formed `AgentResponse` (interpreted intent, safety note,
   recommendations or fallback, score breakdown, optional warnings).
3. **Documentation** — this file is the human-readable summary of
   what each query *should* do.

All Tavily calls in the test suite are mocked through
`tavily_client_factory`. No real network traffic is generated by `pytest`.

## What the universal contract test verifies on every query

Every golden query must:

1. **Generate a well-formed response.** `run_recommendation` returns
   an `AgentResponse` with `query == spec["query"]`.
2. **Populate `interpreted_intent`.** The parsed `AgentIntent` is
   attached to the response.
3. **Always include the safety note.** `resp.safety_note` equals the
   constant `SAFETY_NOTE` exposed by the engine.
4. **Either recommend or fall back.** `resp.recommendations` is
   non-empty *or* `resp.fallback_message` is a non-empty explanation.
5. **Provide a `ScoreBreakdown` on every recommendation.** Each
   `AgentRecommendation` has all 9 score components plus a
   `final_score` clipped to `[0, 1]`.
6. **Respect the state filter.** When the query specifies a state,
   every recommendation's `state` field equals it (unless the
   recommendation carries the `retrieved_via_relaxation` warning,
   which only fires when the retriever explicitly relaxed the
   filter).
7. **Respect the facility_type filter.** Same rule as state, with
   the same relaxation escape hatch.
8. **Carry evidence snippets where supported.** When recommendations
   are returned, the top one has at least one `EvidenceSnippet`
   covering one of the expected capabilities (unless the candidate
   was retrieved via vector-only / relaxation paths, which are
   flagged in `warning_flags`).
9. **Surface warnings for risky results.** GQ-009 and GQ-010 have
   dedicated assertions that check `recommendation_readiness ≠
   "Ready for recommendation"` and that
   `validation_findings` + `warning_flags` are non-empty
   respectively.

A separate parametrized test runs all 10 queries with a mocked Tavily
client (`enable_web_verification=True`, `max_web_verified=2`) and
confirms that:

- The factory is called with the API key.
- `resp.trace_summary["tavily"]["enabled"]` is `True`.
- Every recommendation that was sent for verification carries a
  `WebVerificationResult` whose `verification_status` is one of
  `verified` / `partial` / `unverified`.

No real Tavily traffic is generated by `pytest`.

## Running the suite locally

```bash
pytest tests/test_golden_queries.py -v
```

The suite is fast (≈1.4s on a developer laptop) because the
fixture DataFrame is small, the intent parser is pure regex, and
all external services (Databricks Vector Search, Tavily) are
either mocked or short-circuited via the
`enable_vector_search=False` / `enable_web_verification=False`
defaults.

## Known parser gaps documented by this suite

- **Capability over-extraction.** ~~Substring matches against short
  keywords like `"OR"` (a SURGERY synonym) or `"ER"` (an
  EMERGENCY_TRAUMA keyword) fire on benign substrings such as
  `"f**or**"`, `"supp**or**t"`, `"cent**er**s"`.~~ **Fixed in
  Stage 16.** `agent_core/capability_taxonomy.py` introduces
  `normalize_text` / `term_matches` / `find_matching_terms`, which
  enforce strict word-boundary matching for any term ≤ 3 characters
  or containing `0–9 / x : &`. `find_capabilities_in_text()` was
  rewired through these helpers, so `"Find dialysis centers in
  Uttar Pradesh"` now extracts `DIALYSIS_RENAL` only — no spurious
  `EMERGENCY_TRAUMA` from `"centers"`. The subset semantics on the
  intent test still hold but no longer have any false positives to
  tolerate. Regression coverage in
  [`tests/test_clinical_matching_quality.py`](../tests/test_clinical_matching_quality.py).
- **Verification trigger phrase coverage.** The phrase
  `"need human verification"` is not in the
  `_TRUST_TRIGGERS["verification_ok"]` list (which requires the
  exact substrings `"needs verification"` or `"with verification"`).
  GQ-009 documents this gap and demonstrates that the verification
  intent is still satisfied at result-display time by surfacing
  facilities whose `recommendation_readiness` is non-Ready.
  *Status: still open — deferred from Stage 16.*

## Stage 16 strengthening notes

Stage 16 ("Clinical Matching Quality Patch") tightened the
clinical-match logic that two of the golden queries depend on:

- **GQ-003 (Maharashtra emergency).** The recommendation engine
  now adds a 9th score component `_W_LOCAL_RELEVANCE = 0.10`
  scaled by `local_relevance_score / 15.0`. This is what lets
  `F-MAH-02` (Mumbai Trauma Centre) outrank `F-MUM-01` (Apollo
  Mumbai) on the emergency query — depth of local match now
  influences final ordering, not just trust score.
  `EMERGENCY_TRAUMA`'s `strong_evidence_keywords` were tightened
  to clinically meaningful phrases (`emergency department`,
  `trauma centre`, `ambulance`, `24/7`, `ventilator`, `triage`) so
  generic words like `surgery` / `treatment` no longer satisfy
  emergency validation by themselves.
- **GQ-004 (UP dialysis).** `DIALYSIS_RENAL` lost the bare
  keywords `kidney` and `renal` (which were matching `kidney
  stones` and `adrenal`). Strong evidence now requires specific
  phrasings such as `haemodialysis`, `dialysis machine`, `AV
  fistula`, `peritoneal dialysis`, `CRRT`, or `nephrology unit`.
  `local_retriever.py` adds a `WEIGHT_NAME_MATCH = 3.0` bonus so
  a facility named `"Lucknow Dialysis Centre"` outranks a generic
  clinic that merely mentions `kidney` somewhere in its evidence.
- **GQ-010 (sketchy emergency hospital).** A new contradiction
  rule `CR_HIGH_ACUITY_CLAIM_NO_EVIDENCE` in
  `agent_core/contradiction_rules.py` fires (severity `high`)
  whenever a record claims emergency / trauma / ICU / dialysis /
  critical care in `specialties` or `capabilities_raw` but has
  *both* `equipment` and `procedures` blank — closing a loophole
  where `F-RISKY-01` could otherwise have passed validation as
  "supported" purely on the strength of the word `"24/7"` in its
  capabilities text.
